# AI Agent面试题

# 第一部分：RAG 核心原理

**Q1: 请解释 RAG 的工作原理。与直接对 LLM 进行微调相比，RAG 主要解决了什么问题？有哪些优势？**

## 一、什么是 RAG？

> **先查资料，再回答。**

也就是说，当用户提问时，系统不是只依赖 LLM 参数里“记住”的知识，而是会先去外部知识库中检索与问题最相关的内容（例如文档片段、PDF 段落、数据库记录），再把这些检索到的内容连同用户问题一起交给模型生成答案。

---

## 二、RAG 的基本工作流程

从工作流程上看，RAG 通常可以分成以下几个阶段：

### 1. 知识入库
将文档、PDF、网页、FAQ 等资料切分成较小片段（chunks），并为每个片段生成 embedding（向量表示），存入向量库。

### 2. 用户提问
用户输入一个自然语言问题。

### 3. 检索阶段
系统把用户问题也转成向量，在向量库里查找语义上最相关的若干片段。

这里不是简单的关键词匹配，而是按**语义相似度**进行检索，因此即使问题与原文措辞不同，也有可能找到正确答案。

### 4. 增强阶段
把检索到的相关片段作为上下文，附加到 prompt 中。

### 5. 生成阶段
大语言模型基于：

- 用户问题
- 检索到的上下文材料

生成最终回答。

---

## 三、RAG 相比直接微调主要解决了什么问题？

与**直接微调（Fine-tuning）**相比，RAG 主要解决以下几个关键问题：

---

### 1. 解决“知识更新困难”的问题

如果只依赖微调，模型学到的新知识会被“写进参数里”。

一旦知识库更新（例如政策变化、产品文档改版、论文有新版本），通常就需要：

- 重新准备数据
- 重新训练
- 或再次微调

这不仅成本高，而且流程繁琐。

而 RAG 把知识放在**外部知识库**中，只要更新文档或向量库，模型下次检索时就能直接使用最新内容，而**不需要重新训练模型**。

这对于以下类型的知识尤其重要：

- 公司内部文档
- 法律条文
- 产品信息
- 最新业务规则

---

### 2. 解决“模型不知道你的私有数据”的问题

通用 LLM 并不知道你自己的：

- PDF 文档
- 内部手册
- 客户资料
- 项目文档

虽然微调可以让模型“学到一些”这类内容，但训练过程复杂，而且并不适合频繁变化的私有资料。

RAG 则可以直接接入你的私有文档库，在查询时临时检索出相关片段，因此特别适合：

- 企业内部知识库
- 个人文档 / PDF
- 法律、医疗、金融等专业资料
- 客服 FAQ、产品说明书

---

### 3. 解决“回答缺乏依据、难以溯源”的问题

单纯依赖微调或模型内部记忆时，模型也许能回答问题，但你往往很难判断它是：

- 真正基于可靠信息回答
- 还是“猜出来的”

RAG 的优势在于：答案是基于检索到的文档片段生成的，因此更容易附上引用、来源或命中文档信息。

这带来两个直接好处：

- **更可验证**：可以检查答案依据的是哪一段文档
- **更可控**：可以限制模型“只能根据检索内容回答”

这在高可靠性场景中尤其重要，例如：

- 合同问答
- 制度问答
- 技术支持

---

## 四、RAG 相比微调的主要优势

### （1）知识更新更快、更便宜
更新知识时，通常只需要更新文档和向量库，而不需要重新训练模型。  
相比反复微调，这种方式更加灵活，成本也更低。

### （2）更适合接入外部和私有知识
RAG 天然适合把“模型之外”的数据接进来，尤其适用于：

- 文件
- 数据库
- 企业文档
- 私有知识库

### （3）可解释性更强
因为答案建立在可检索到的上下文之上，通常能够给出来源依据，而不是一个完全黑箱式的回答。

### （4）降低幻觉风险
如果检索到的上下文质量较高，模型更容易围绕真实材料作答，而不是凭空“脑补”。

> 注意：这里是**降低**幻觉，而不是**彻底消除**。  
> 如果检索结果本身有误、检索不完整，或者模型过度发挥，幻觉仍然可能发生。

### （5）不必把所有知识都“塞进模型参数”
微调更像是在改造模型的行为或风格；  
RAG 更像是给模型配备了一个“可随时查阅的外部记忆”。

对于大规模、频繁更新的知识，RAG 通常更实际。

---

## 五、那微调还有什么价值？

有。  
**RAG 并不是替代微调，而是与微调解决不同层面的问题。**

### RAG 更擅长解决
> 模型需要访问**最新的、外部的、私有的事实知识**。

### 微调更擅长解决
> 模型需要形成特定的**输出风格、格式、任务习惯、领域表达方式**。

## 六、具体例子

### 适合微调的场景
如果你希望模型能够：

- 使用公司统一的客服语气
- 按固定 JSON 格式输出
- 遵循某种审稿风格

那么微调会很有帮助。

### 适合 RAG 的场景
如果你希望模型能够回答：

- 最新版员工手册第 7 条是什么
- 某份 PDF 中如何定义某个概念

那么 RAG 会更合适。

生成阶段：LLM 基于“问题 + 检索结果”组织答案。
